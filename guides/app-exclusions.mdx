---
title: App Exclusions
description: Disable Neru in specific applications using bundle IDs
---

App exclusions allow you to prevent Neru from activating in specific applications. This is useful for terminal apps, remote desktop clients, or any application where you prefer keyboard-only workflows.

## Overview

When an app is excluded, Neru will:
- Not respond to hotkeys while that app is active
- Ignore CLI commands to enter navigation modes
- Remain running in the background

<Note>
  Excluded apps are configured per bundle ID, not by application name. This ensures reliable identification across app updates and renames.
</Note>

## Configuration

Add bundle IDs to the `excluded_apps` array in your config:

```toml
[general]
excluded_apps = [
    "com.apple.Terminal",
    "com.googlecode.iterm2",
    "com.microsoft.rdc.macos",
]
```

<ParamField path="general.excluded_apps" type="array" default="[]">
  List of bundle IDs where Neru won't activate. Empty array means no exclusions.
</ParamField>

## Finding Bundle IDs

### Method 1: Using osascript (macOS)

Get the bundle ID of any running application:

```bash
osascript -e 'id of app "Safari"'
# Output: com.apple.Safari

osascript -e 'id of app "Visual Studio Code"'
# Output: com.microsoft.VSCode
```

### Method 2: Using mdls (macOS)

Get bundle ID from application path:

```bash
mdls -name kMDItemCFBundleIdentifier -r /Applications/Safari.app
# Output: com.apple.Safari
```

### Method 3: Info.plist

Manually inspect the Info.plist file:

```bash
defaults read /Applications/Safari.app/Contents/Info CFBundleIdentifier
# Output: com.apple.Safari
```

### Method 4: Activity Monitor

1. Open **Activity Monitor** (Applications → Utilities)
2. Find your application in the list
3. Double-click the process
4. Look for **Bundle Identifier** in the process info

## Testing Exclusions

After adding exclusions:

1. Reload configuration:
   ```bash
   neru config reload
   ```

2. Focus the excluded app

3. Try triggering Neru with a hotkey (e.g., `Cmd+Shift+Space`)

4. Verify that Neru does not activate

<Note>
  If Neru still activates, double-check the bundle ID with `osascript -e 'id of app "AppName"'`
</Note>

## Dynamic Exclusions

Currently, Neru does not support dynamic exclusions (enabling/disabling apps at runtime). To modify exclusions:

1. Edit your config file
2. Run `neru config reload`

Alternatively, restart the daemon:

```bash
pkill neru && neru launch
```

## Alternative: Per-app Hide Overlay

For screen sharing apps specifically, consider using the screen share toggle instead:

```toml
[general]
hide_overlay_in_screen_share = true
```

Or toggle dynamically:

```bash
neru toggle-screen-share
```

See [Screen Sharing](/guides/screen-sharing) for more details.

## Troubleshooting

<AccordionGroup>
  <Accordion title="App not being excluded">
    **Check:**
    - Bundle ID is correct (verify with `osascript -e 'id of app "AppName"'`)
    - Config has been reloaded: `neru config reload`
    - App is actually focused (not a different window)
    
    **Verify loaded config:**
    ```bash
    neru config dump | grep excluded_apps
    ```
  </Accordion>
  
  <Accordion title="Neru not working in any app after exclusion">
    **Check:**
    - You didn't accidentally exclude all apps
    - No typos in bundle IDs
    - Config syntax is valid (TOML array format)
    
    **Valid syntax:**
    ```toml
    [general]
    excluded_apps = [
        "com.apple.Terminal",  # Comma-separated
        "com.googlecode.iterm2",
    ]
    ```
  </Accordion>
  
  <Accordion title="Exclusion not working after app update">
    **Sometimes apps change bundle IDs after updates.**
    
    **Solution:**
    1. Get the new bundle ID:
       ```bash
       osascript -e 'id of app "AppName"'
       ```
    2. Update your config with the new ID
    3. Reload: `neru config reload`
  </Accordion>
</AccordionGroup>
