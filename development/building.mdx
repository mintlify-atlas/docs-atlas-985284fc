## Development Environment Setup

Choose one of the following setup methods:

### Option A: Devbox (Recommended)

[Devbox](https://www.jetify.com/devbox) provides an isolated development environment with all required tools pre-configured.

<Steps>
  <Step title="Install Devbox">
    ```bash
    curl -fsSL https://get.jetify.com/devbox | bash
    ```
  </Step>
  
  <Step title="Enter the development shell">
    **Option 1**: Manual activation
    ```bash
    devbox shell
    ```
    
    **Option 2**: Automatic activation with direnv (recommended)
    ```bash
    # Install direnv
    brew install direnv
    
    # Add to your shell config (~/.bashrc, ~/.zshrc, or ~/.config/fish/config.fish)
    eval "$(direnv hook bash)"  # or zsh/fish
    
    # The .envrc file will automatically activate devbox when you cd into the project
    ```
  </Step>
</Steps>

Devbox automatically installs and manages:
- Go 1.26.0
- gopls (Go language server)
- gotools, gofumpt, golines (Go formatting tools)
- golangci-lint (linter)
- just (command runner)
- clang-tools (C/C++ tools for CGo)

### Option B: Manual Installation

Install essential tools manually using Homebrew:

```bash
brew install go just golangci-lint llvm
```

**Optional additional tools** (install via `go install` if desired):

```bash
go install golang.org/x/tools/gopls@latest
go install mvdan.cc/gofumpt@latest
go install github.com/segmentio/golines@latest
```

## Build Commands

Neru uses [Just](https://github.com/casey/just) as a command runner (alternative to Make).

### Common Build Tasks

| Task   | Command                 | Description                        |
| ------ | ----------------------- | ---------------------------------- |
| Build  | `just build`            | Development build (auto-detects version from git tags) |
| Test   | `just test`             | Run unit tests                     |
| Test   | `just test-integration` | Run integration tests              |
| Test   | `just test-all`         | Run all tests (unit + integration) |
| Test   | `just test-race`        | Run tests with race detection      |
| Lint   | `just lint`             | Run linters                        |
| Format | `just fmt`              | Format code                        |
| Run    | `just run`              | Build and run the application      |
| Clean  | `just clean`            | Remove build artifacts             |

### Build Variants

<Tabs>
  <Tab title="Development">
    ```bash
    # Development build
    just build
    
    # Build with custom version
    just build-version v1.0.0
    ```
    
    Creates a debug build with version info from git tags.
  </Tab>
  
  <Tab title="Release">
    ```bash
    # Release build (optimized, stripped)
    just release
    ```
    
    Creates an optimized build with debug symbols stripped.
  </Tab>
  
  <Tab title="Bundle">
    ```bash
    # Build macOS app bundle
    just bundle
    ```
    
    Creates a macOS `.app` bundle for distribution.
  </Tab>
</Tabs>

### Manual Build (Without Just)

If you prefer not to use Just:

```bash
# Basic build
go build -o bin/neru ./cmd/neru

# With version info
VERSION=$(git describe --tags --always --dirty)
go build \
  -ldflags="-s -w -X github.com/y3owk1n/neru/internal/cli.Version=$VERSION" \
  -o bin/neru \
  ./cmd/neru

# For release (optimized)
go build \
  -ldflags="-s -w -X github.com/y3owk1n/neru/internal/cli.Version=$VERSION" \
  -trimpath \
  -o bin/neru \
  ./cmd/neru
```

### Build Flags Explained

- `-ldflags="-s -w"`: Strip debug info and symbol table (smaller binary)
- `-trimpath`: Remove file system paths from binary
- `-X pkg.Var=value`: Set string variable at build time (version injection)

## Testing

Neru has a comprehensive test suite with clear separation between unit tests and integration tests.

### Test Organization

| Test Type             | File Pattern              | Purpose                                                            | Command                 |
| --------------------- | ------------------------- | ------------------------------------------------------------------ | ----------------------- |
| **Unit Tests**        | `*_test.go`               | Business logic with mocks (no build tag required)                  | `just test`             |
| **Integration Tests** | `*_integration_*_test.go` | Real system interactions (tagged `//go:build integration && <os>`) | `just test-integration` |

### Running Tests

<Tabs>
  <Tab title="Unit Tests">
    ```bash
    # Run all unit tests (fast, CI)
    just test
    
    # Test specific package
    go test ./internal/core/domain/hint/
    
    # Test with verbose output
    go test -v ./internal/app/services/
    ```
  </Tab>
  
  <Tab title="Integration Tests">
    ```bash
    # Run all integration tests (comprehensive, local)
    just test-integration
    
    # Integration test specific component
    go test -tags=integration ./internal/core/infra/accessibility/
    ```
  </Tab>
  
  <Tab title="All Tests">
    ```bash
    # Run all tests (unit + integration)
    just test-all
    
    # With race detection
    just test-race
    ```
  </Tab>
</Tabs>

### Test Coverage

```bash
# Generate coverage report
just test-coverage

# View coverage in browser
go tool cover -html=coverage.out
```

## Debugging

To debug Neru during development:

### Enable Debug Logging

Edit your config file (`~/.config/neru/config.toml`):

```toml
[logging]
log_level = "debug"
```

### View Logs

<Tabs>
  <Tab title="macOS">
    ```bash
    tail -f ~/Library/Logs/neru/app.log
    ```
  </Tab>
  
  <Tab title="Linux">
    ```bash
    tail -f ~/.local/state/neru/log/app.log
    ```
  </Tab>
</Tabs>

### Use Delve Debugger

```bash
# Install Delve
go install github.com/go-delve/delve/cmd/dlv@latest

# Debug Neru
dlv debug ./cmd/neru
```

## Cross-Platform Setup

Neru can be developed on any OS, but some features require platform-specific APIs.

- **macOS**: Full environment support (CGo, Accessibility, Overlays)
- **Linux**: Requires `gcc` and `pkg-config` for build foundations. AT-SPI and X11/Wayland development tools are recommended.
- **Windows**: Requires a C compiler (e.g., MinGW or MSVC) for CGo components. UI Automation (UIA) SDK is recommended.

## Troubleshooting

### Build Failures

<AccordionGroup>
  <Accordion title="CGo compilation errors">
    Ensure Xcode Command Line Tools are installed:
    ```bash
    xcode-select --install
    ```
  </Accordion>
  
  <Accordion title="Missing dependencies">
    Update Go modules:
    ```bash
    go mod tidy
    go mod download
    ```
  </Accordion>
  
  <Accordion title="Version detection issues">
    Ensure you're in a git repository with tags:
    ```bash
    git fetch --tags
    git describe --tags --always --dirty
    ```
  </Accordion>
</AccordionGroup>

### Test Failures

- **Integration tests failing**: Ensure you have the required platform APIs (macOS Accessibility permissions)
- **Race detector warnings**: These indicate potential concurrency issues that should be addressed
